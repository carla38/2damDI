Proceso Resolución - Monitorización Servidores

1.- Introducción

 

El objetivo de esta práctica ha sido crear un Dashboard para el control de servidores, que sea capaz de iniciar y detener los servidores y de mostrar su estado actual y otras características. Estos servidores serán Máquinas Virtuales creadas en VirtualBox.

El proyecto en su mayoría se ha realizado usando AntiGravity para generar los distintos archivos para el frontend y backend, además de scripts para una activación directa del backend tanto en Windows como Linux y una versión portable para Windows. También se ha usado ChatGPT como apoyo para realizar algunos prompts y pasos de configuración y preparación que había que realizar manualmente o que la IA de AntiGravity no generaba correctamente.

Se irán documentando por apartados las diferentes partes que conforman el proyecto, su contenido y funcionamiento.

 

 

2.- VirtualBox

 

Lo primero que hice y lo más sencillo fue preparar algunas Máquinas Virtuales en VirtualBox. Este proyecto está pensado para que el backend detecte directamente todas las VMs que contiene, por lo que no ha sido necesario hacer ninguna configuración extra en las VMs.

4.- Backend

 

El Backend ha sido el aspecto de la práctica sobre el que más se ha trabajado y más cambios ha habido. En un principio se usaba SSH, configurándolo en cada VM para la conexión remota con el Frontend, pero la configuración y las pruebas resultaban más complicadas, así que se pasó a usar VBoxManage, interfaz de comandos de VirtualBox, para conectar el backend directamente a las VM del host.

La estructura de la carpeta Backend en su versión final se ve así:

-__pycache__ : Carpeta creada automáticamente para la ejecución de Python.

-venv : Entorno virtual de Python.

-requirements.txt : Lista de dependencias de Python para instalar en venv.

Veremos más a fondo los archivos main.py y vbox_manager.py

--main.py--

Es el archivo principal del Backend y el que se encarga de comunicar el Frontend con VirtualBox a partir de vbox_manager

  • Importaciones: 
    • FastAPI, para crear la API del Backend
    • CORS, para comunicar el frontend con la API
    • Pydantic, para validar datos de entrada
    • vbox_manager, lógica para controlar y consultar las VMs
  • Crea la instancia de la API y habilita el CORS para el frontend. 
  • Endpoint "/". Comprueba que el Backend esté activo.
  • Endpoint "/vms". Devuelve una lista con todas las VMs que haya en VirtualBox. El Frontend lo usa para crear las tarjetas de las VMs. 
  • Endpoints "/vms/{name}/start" y "/vms/{name}/stop". Llaman a funciones en vbox_manager.py que se encargan de ejecutar los comandos de VirtualBox que inician y apagan las VMs. 
  • Endpoint "/vms/{name}/details". Devuelve los detalles de las VMs. El Frontend lo usa para mostrar la información en las tarjetas. 

 

--vbox_manage.py--

Este archivo es el que conecta el Backend con VirtualBox, proporcionando toda la información relacionada con las VMs de VirtualBox y el que permite arrancarlas y demás acciones.

Partes a destacar de este código son:

  • Busca VBoxManage en el dispositivo. 
  • Si no ha encontrado VBoxManage, crea unas VMs de prueba. No son funcionales, pero permite visualizar como sería la aplicación. 
  • run_command. Ejecuta VBoxManage y maneja error y tiempo de espera.
  • list_vms. Devuelve las VMs y su estado actual. 
  • start_vm y stop_vm. Para arrancar y detener las VMs. 
  • get_vm_details. Devuelve la información sobre las VMs. 

 

Como resumen, el flujo entre los componentes del proyecto sería el siguiente:

VirtualBox --> vbox_manager --> main.py (Backend) --> script.js (FrontEnd) --> index.html (Dashboard)

--Activación por consola--

En las primeras versiones había que introducir comandos por consola para activar el Backend. Este un ejemplo de este tipo de ejecución (Windows):

Primero habría que ir al directorio del Backend. Con venv\Scripts\activate se activaría el entorno virtual, en el que se descargarían las dependencias con pip install -r requirement.txt.

--Versión Portable--

Al acabar se me ocurrió hacer una versión portable del proyecto, con la idea de que además fuera más sencillo llevarlo al aula, ya que esta versión no requerirá que el dispositivo tenga instalado Python y demás dependencias. 

Antigravity no podía generar la versión portable del Backend, asi que seguí las pautas que me dió ChatGPT para crearlo. Primero cree una carpeta 'portable', que contenía a su vez una carpeta para el Frontend (con exactamente el mismo contenido que la carpeta local) y otra para el Backend.

Después, desde la consola, se instala Pyinstaller, que permitirá empaquetar Python junto a sus dependencias y el backend, generando un ejecutable que no requerirá instalación. Para instalarlo se usa pip install pyinstaler desde el entorno virtual.

Una vez instalado, con el comando pyinstaller --onefile --name dashboard-backend backend/main.py se creará una carpeta con un .exe en la raíz de proyecto, el cuál habrá que mover a la carpeta Backend del portable.
